# Introduction:
- Most applications are built by layering one data model on top of another.
- **The Core Rule:** Each layer hides the complexity of the layers below it by providing a clean data model. 
	- *Example:* Data Structures (App Level) $\rightarrow$ Data Model (DB Level) $\rightarrow$ Bytes (OS/Memory Level) $\rightarrow$ Electrical/Magnetic signals (Hardware Level).

# Relational Model vs. Document Model:
## The Birth of NoSQL:
- **NoSQL** originally meant "non-relational," but is now retroactively interpreted as *Not Only SQL*
- **Driving forces behind NoSQL adoption (The 2026 Architectural Matrix):**
	- **1. Need for greater scalability (very large datasets or very high write throughput):**
		- **Where SQL Caught Up:** The rise of "NewSQL" / Distributed SQL (e.g., Google Spanner, CockroachDB, Yugabyte). They achieved horizontal sharding and high availability without sacrificing ACID transactions.
		- **Where SQL Needs to Catch Up:** Pure, extreme-throughput write latency. Distributed SQL still pays the network penalty of Two-Phase Commit (2PC) for consensus. For ultra-fast, uncoordinated "fire-and-forget" event ingestion, leaderless NoSQL (like Cassandra) still wins.
	- **2. Widespread preference for free and open-source software:**
		- **Where SQL Caught Up:** PostgreSQL and MySQL absolutely dominated this space. The days of needing million-dollar Oracle licenses for enterprise-grade relational features are long dead. 
		- **Where SQL Needs to Catch Up:** This gap is essentially closed. However, NoSQL ecosystems originally moved faster on developer-friendly "Serverless DBaaS" (like DynamoDB), though Serverless Postgres (Aurora, Neon) has heavily narrowed this gap.
	- **3. Specialized query operations not well-supported by relational models:**
		- **Where SQL Caught Up:** Extensibility. Postgres added Recursive CTEs (for graphs), extensions like TimescaleDB (for time-series), and `pgvector` (for AI/embedding searches).
		- **Where SQL Needs to Catch Up:** Extreme-scale specialization. A Postgres Recursive CTE is still $O(N)$ and will melt on a 6-hop graph query, whereas Neo4j ($O(1)$) will return it instantly. Postgres `tsvector` is great, but it will never replace Elasticsearch for complex, fuzzy, relevancy-scored full-text search across terabytes of data.
	- **4. Frustration with the restrictiveness of relational schemas (desire for dynamic data models):**
		- **Where SQL Caught Up:** `JSONB` in Postgres. You can now dump unstructured, nested data directly into a SQL table and index it using GIN indexes, achieving "schema-on-read" inside a relational database.
		- **Where SQL Needs to Catch Up:** Disk I/O efficiency on nested updates. If you update a single deeply nested key in a Postgres `JSONB` column, the engine often has to rewrite the entire JSON object on disk. Document databases (MongoDB) are better optimized for in-place, sub-document mutations without rewriting the whole tree.
- **Polyglot Persistence:** idea of use of relational db alongside non-relational db

## The Object-Relational Impedance Mismatch:
- Most application code is object-oriented, but relational databases use tables, rows, and columns. The disconnect between these two models requires a translation layer, known as the **impedance mismatch**.
- ORM (Object-Relational Mapping) frameworks like *Hibernate* or *ActiveRecord* reduce boilerplate code, but act as a **leaky abstraction** under high performance or scale requirements.
	- **Core Structural Failure Points of ORMs:**
		- **1. The N+1 Query Trap:** In OO code, traversing a relationship (e.g., `user.getJobs()`) looks like an $O(1)$ memory operation. In reality, looping over a list of 100 loaded objects can trigger 100 sequential, synchronous network queries to the database instead of a single optimized `JOIN`.
		- **2. The Inheritance Nightmare:** Relational tables do not support object inheritance. Mapping inheritance hierarchies forces poor trade-offs: either a single table packed with sparse `NULL` columns, or multiple joined tables requiring heavy, slow implicit `JOIN` operations for every look-up.
		- **3. The Analytics & Aggregation Wall:** ORMs are engineered around single-entity tracking. Complex analytical reporting—such as window functions (`RANK() OVER (...)`), Common Table Expressions (CTEs), or heavy multi-dimensional grouping—becomes unreadable or completely unsupported in ORM query abstractions (like JPQL).
		- **4. Dirty Checking Memory Inflation:** Heavy ORMs must retain an exact snapshot of loaded entities in memory (L1 cache/Persistence Context) to figure out mutations for `UPDATE` statements. Processing large batch records causes massive JVM heap overhead, triggering major Garbage Collection spikes.
	- **Modern Architectural Remedies (The 2026 Standard):**
		- **Micro-ORMs & Pure Query Builders:** Shifting away from heavy object-tracking mappers toward tools like **jOOQ**, **MyBatis**, or **sqlc**. These tools treat SQL as a **first-class citizen**, maintaining type safety while returning raw control over query compilation back to the engineer.
		- **Strategic JSONB Ingestion:** Instead of building a complex multi-table relational schema for highly volatile, nested data or deeply branched one-to-many properties (like dynamic user setting matrices), engineers map a single `JSONB` column directly to native code maps, bypassing object-relational mapping completely.
		- **CQRS (Command Query Responsibility Segregation):** Physically decoupling mutations from reads to stop database architecture wars:
			- **The Command Side (Writes):** Uses normalized relational tables optimized for safe, validated, high-integrity ACID transactions. Read performance is ignored here.
			- **The Query Side (Reads):** Bypasses the write engine and ORM entirely. Fed by an asynchronous **Change Data Capture (CDC)** engine (e.g., Debezium) streaming events via **Kafka** into read-optimized datastores like **Elasticsearch** or **Redis**. Data is stored completely denormalized and pre-shaped exactly as the client UI expects it.
			- **The Architectural Trade-off:** Resolving the read/write lock conflict requires accepting **Eventual Consistency**. Systems use optimistic UI rendering or state-forwarding techniques to mask consumer replication lag from the user.
- **The Document Model Advantage:** JSON reduces the object-relational impedance mismatch by naturally representing one-to-many relationships as an explicit tree structure (e.g., a User profile with nested Job and Education arrays). This provides excellent **storage locality**, allowing the entire tree to be retrieved in a single disk seek/database query.
	- **The Core Trade-off: Hidden Flaws of JSON Encoding:**
		- **1. Number Precision Risks:** JSON lacks distinct integer/float types and defaults to IEEE 754 double-precision floats. This silently truncates/rounds 64-bit integers (like Snowflake IDs) above $2^{53}$ unless explicitly handled as strings (e.g., Twitter API's `id_str`).
		- **2. CPU Parsing Tax:** As a plain-text format, JSON serialization and deserialisation force the CPU to scan strings character-by-character. Under intense scale (e.g., 100k+ RPS), string parsing often consumes more compute cycles than core business logic.
		- **3. Payload Verbosity (Network Bloat):** JSON is completely self-describing, meaning exact structural keys (e.g., `"firstName"`, `"lastName"`) are repeated identically across every single object in an array, wasting massive network bandwidth.
		- **4. Binary Inefficiency:** JSON cannot natively encode raw binary data (images, hashes). Forcing binary into JSON requires Base64 conversion, inflating the payload size by **33%**.
	- **Modern Mitigations (The 2026 Production Standard):**
		- **At the Storage Layer:** Production engines bypass raw text. MongoDB translates records into **BSON** (Binary JSON) to introduce rigid type wrappers and faster pointer traversal. PostgreSQL utilizes **JSONB**, a decomposed binary format that completely strips whitespace and accelerates indexing.
		- **At the Network Layer:** For high-throughput internal microservice networks (RPC), teams swap text-based JSON/REST entirely for binary transport mechanisms like **gRPC/Protocol Buffers (Protobufs)** to enforce rigid schemas, eliminate key repetition, and eliminate parsing overhead.

## Many-to-One and Many-to-Many Relationships:
- **One-to-Many Relationships:** Document models handle these beautifully via nested records.
- **Many-to-One & Many-to-Many Relationships:** This is where the document model struggles.
	- Normalizing data requires many-to-one relationships (which means relying on IDs and Joins).
	- Relational databases are built for joins. 
	- Document databases have weak support for joins. If you need a join, you must emulate it in the application code by making multiple queries to the database.
- **The Trap:** If your application uses many-to-many relationships, the document model becomes less appealing. You can try to avoid joins by denormalizing data (duplicating it), but then the application code has to do extra, complex work to keep the duplicated data consistent.
- In a relational database, the **Query Optimizer** automatically decides which parts of the query to execute in which order, and which indexes to use.
- There is no universal winner; it depends entirely on the shape of your data:
	- **The Document Win (The Tree Structure):** If your application data is naturally shaped like a document (a self-contained tree, like a resume loaded all at once), the document model simplifies code. Relational databases force you to perform **"shredding"**—splitting a single logical document into multiple tables (users, jobs, education), which requires cumbersome schema design and complex `JOIN` queries to reassemble.
	- **The Relational Win (The Interconnected Web):** If your application requires many-to-many relationships, the document model makes application code significantly *more complex*. Because document DBs have weak/no support for joins, developers are forced to choose between two bad options:
		- **1. App-Side Joins:** Making multiple sequential network requests to the database to manually emulate a `JOIN` in the application code (slow and complex).
		- **2. Denormalization:** Duplicating data across documents to avoid joins. The application code now has to take on the massive burden of keeping all that duplicated data perfectly synchronized and consistent upon every update.

## Relational Versus Document Databases Today:
- **Schema Flexibility: Schema-on-Read vs. Schema-on-Write:**
	- Document databases are often falsely marketed as "schemaless." A more accurate comparison is how programming languages handle types.
	- **Schema-on-Read (Document):** Similar to **dynamic type checking**. The schema is implicit. The database allows arbitrary data, and the application code interprets the structure when reading it.
	- **Schema-on-Write (Relational):** Similar to **static type checking**. The schema is explicit. The database is the gatekeeper and guarantees that all written data conforms to the rules. 
		- **The Schema Migration Myth:** SQL migrations have a bad reputation for causing downtime, largely due to historical quirks (e.g., older MySQL versions copying the entire table for an `ALTER TABLE`).
		- **The Reality:** In modern DBs like PostgreSQL, `ALTER TABLE ADD COLUMN` is virtually instantaneous because it only updates schema metadata.
		- **The Real Bottleneck:** The true I/O penalty comes from the `UPDATE` statement. If you add a column and run a script to backfill 50 million existing rows, the database must physically rewrite every row on the disk, crushing performance.
		- **The Architectural Hack:** To avoid the `UPDATE` penalty in a massive SQL table, engineers add the new column with a default of `NULL`. The application code is then written to check if the value is `NULL` and handle it dynamically. **This effectively uses a NoSQL "Schema-on-Read" strategy inside a strict relational database.**
- **Data Locality and The I/O Penalty:**
	- **The Locality Advantage:** Document DBs store data as a single continuous string (like BSON). If your application needs the *entire* document at once, this is incredibly fast (a single disk seek).
	- **The I/O Trap:** This locality is a double-edged sword:
		- **Read Waste:** The database must load the entire document into memory even if you only query a single small field.
		- **Write Penalty (In-Place vs. Out-of-Place Updates):**
			- **The Easy Op (In-Place):** If an update does not increase the physical byte size of the document (e.g., updating a counter from 10 to 11, or flipping a boolean), the DB simply overwrites the existing memory block. It is highly performant and requires no index pointer changes.
			- **The Hard Op (Out-of-Place):** If an update adds new data (e.g., pushing an object to a nested array) and exceeds the assigned memory footprint, the DB cannot update it in place. It must find a new, larger location on the disk, copy the entire document over, update the B-Tree index, and leave the old location as fragmented "dead space."
		- **Best Practice:** Document databases are only highly performant if documents are kept relatively small and avoid unbounded growth operations (like an infinitely growing `logs` array). 
	- **The 2026 Reality Check (MVCC):** Be aware that modern engines (like PostgreSQL and MongoDB's WiredTiger) use Multi-Version Concurrency Control (MVCC). Under MVCC, *all* updates are technically written out-of-place as new versions to allow safe concurrent reads without locking. However, the architectural law remains: schemas that force documents to grow endlessly will still thrash your disk I/O and bloat your storage.
	- *Note:* Locality is not exclusive to NoSQL. Relational Spanner uses interleaved tables, and Cassandra uses column-families to achieve similar physical disk locality.

# Query Languages for Data:
## Declarative vs. Imperative Query Languages:
- **Imperative (e.g., standard programming languages):** You tell the computer *how* to perform the operation. You write the loops, evaluate the conditions, and manage the variables step-by-step.
- **Declarative (e.g., SQL, Relational Algebra):** You specify *what* pattern of data you want (conditions, groupings, sorts), but not *how* to retrieve it. 
	- *Example:* `SELECT * FROM animals WHERE family = 'Sharks'`
- **The Architectural Advantage of Declarative:**
	- **1. Optimizer Freedom:** It hides the implementation details. The database's query optimizer is free to completely change the execution order, choose different indexes, or upgrade its internal algorithms without breaking your application code.
	- **2. Parallelization:** Imperative loops are strictly sequential and hard to parallelize. Declarative queries just state the goal, allowing the database engine to easily split the work across multiple CPU cores or distributed machines.

## MapReduce Querying (The Distributed Processing Shift):
- **MapReduce:** A programming model popularized by Google for processing massive amounts of data in bulk across a cluster of machines. 
- It relies on two pure functions (meaning they cannot have side-effects or perform external database queries):
	- **1. Map (Collect):** Filters and sorts data, emitting key-value pairs.
	- **2. Reduce (Fold/Aggregate):** Groups and processes those emitted pairs.
- **The Trap:** Writing two perfectly coordinated JavaScript functions to perform a simple group-by query is difficult and error-prone. It is a "low-level" abstraction.
- **The Evolution:** Because of this friction, databases like MongoDB introduced **Aggregation Pipelines**—a declarative, JSON-based query language that provides the power of MapReduce without forcing the engineer to write manual execution loops.

# Graph-Like Data Models:
## Graph-Like Data Models (Handling Extreme Interconnectivity):
- If your application has mostly one-to-many relationships (a tree), use a **Document Model**. If relationships are simple, use a **Relational Model**.
- If your data has highly complex, deeply interconnected many-to-many relationships, use a **Graph Model**.
- Graphs consist of two objects: **Vertices** (nodes/entities) and **Edges** (relationships/arcs).
- **Crucial Feature:** Graphs naturally handle **heterogeneous data**. A single graph can store completely different types of objects (People, Locations, Events, Check-ins) and query their connections instantly. 

## Graph Query Languages vs. SQL:
- **Cypher:** A declarative query language built for Property Graphs (like Neo4j). It uses ASCII-art style syntax to visually represent paths.
	- *Syntax Example:* `(Lucy) -[:BORN_IN]-> (Idaho)` 
- **The SQL Deficit:** Relational databases *can* query graphs using Recursive Common Table Expressions (CTEs) like `WITH RECURSIVE`. However, navigating a complex web of relationships in SQL is incredibly verbose and slow. A query that takes 4 lines of clean Cypher code might require 29 lines of complex, hard-to-maintain SQL. 

## Triple-Stores & SPARQL:
- **The Model:** An alternative to Property Graphs. It stores all data as three-part statements: `(Subject, Predicate, Object)`. 
- **The Two Types of Objects:** The "Object" part of the triple determines if it is a property or a relationship.
	- *Property Example:* `(Lucy, age, 33)` -> "33" is a primitive literal.
	- *Edge Example:* `(Lucy, marriedTo, Alain)` -> "Alain" is a reference to another vertex.
- **SPARQL Query Example:** The query language uses variables (prefixed with `?`) to pattern-match against these triples to find answers. 
	```sparql
	SELECT ?personName WHERE {
	  ?person :marriedTo :Alain .
	  ?person :name ?personName .
	}
	```

# Summary:
## The "No Silver Bullet" Rule:
- **The Big Three:** Document, Relational, and Graph models are all widely used today because they solve fundamentally different problems.
	- **Document:** Best for self-contained, one-to-many tree structures where external relationships between documents are rare.
	- **Graph:** Best for the exact opposite—use cases where anything is potentially related to everything.
	- **Relational:** The battle-tested middle ground for structured data and standard many-to-one/many-to-many joins.
- **The Emulation Trap:** You *can* emulate one model inside another (e.g., storing a graph inside PostgreSQL using a massive `edges` table), but the result is almost always awkward, slow, and hard to maintain. You must pick the storage engine that naturally fits the shape of your application data.
- **The Schema Reality:** Whether a database uses an explicit schema (enforced on write) or an implicit schema (assumed on read), your application code *always* assumes a structure. The only architectural choice is whether the database or your application is acting as the gatekeeper of that structure.
