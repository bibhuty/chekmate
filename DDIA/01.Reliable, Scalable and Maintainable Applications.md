# Data-intensive Application:
- Data is the primary challenge in the form of(as opposed to compute-intensive, where CPU cycles are bottleneck) :
	- quality of data
		- Same iPhone listed by different sellers as "iPhone 15 Pro," "Apple iPhone15 PRO," and "iphone-15-pro-max" — search and pricing break due to inconsistent, messy input.
	- complexity of data
		- Facebook's "friends of friends who liked similar pages" query requires traversing a deeply connected graph of users, posts, reactions, and groups — not a simple table lookup.
	- speed at which the data is changing
		- Uber must continuously refresh the GPS positions of hundreds of thousands of drivers every few seconds to match riders to the nearest available car in real time.
- A data intensive system is built from standard building blocks i.e. data systems that provide commonly needed functionalities. e.g.:
	- *database* - for storage and retrieval
	- *caches* - speed up reads
	- *search indexes* - flexible search
	- *stream processing* - asynchronous handling of data
	- *batch processing* - periodic crunching of data
- Factors influencing design of a data system includes:
	- skill and experience of people involved
	- legacy system dependency
	- the time scale of delivery
	- orgs tolerance to difference kind of risks
	- regulatory 

# Reliability:
- Expectation from a software includes:
	- performs the function the user expects
	- tolerate the user making mistakes or using the software in unexpected ways
	- performance is good enough for the required use case, under the expected load and data volume
	- system prevents any unauthorised access and abuse
- **Faults:** The thing that can go wrong. It is usually defined as one component of a system deviating from its spec.
- **Fault-tolerant or resilient:** System that expect that faults of certain kind will happen eventually and prepare for them. e.g. fault tolerant DB have replicas
- **Failure:** System as a whole stops providing the required service to the user
- In general its preferred to tolerate faults over preventing, there are cases where prevention is better than cure. e.g. security systems, where preventing an attacker is essential.
## Hardware Faults:
- One option to tolerate hardware faults is adding hardware redundancy to individual hardware components:
	- disks may be set up in a RAID(Redundant Array of Independent Disk) configuration.
		- RAID = storing data across multiple disks using replication, sharding, or a mix — so that one disk failure doesn't bring down the system.
	- servers may have dual power supplies and hot swappable CPUs
	- datacenters may have batteries and diesel generators for backup power
- One more option is to use software fault tolerant techniques like:
	- replication
	- failover
		- automatically redirect traffic to healthy machine if the current machine becomes unhealthy
	- rolling upgrades
	- retries and timeouts
	- checkpointing
	- graceful degradation
	- circuit breaking
		- stopping the system for a while to recover and throwing error to the upstream systems
	- erasure coding
		- instead of storing full duplicate copies of the data (expensive), you store the original data + small mathematical "clues" (compact). If something is lost, you use the surviving data + clues to reconstruct what's missing.

## Software Errors:
- The software is making some kind of assumption about its environment and while that assumption is usually true, it eventually stops being true for some reason. These are the errors during development stage.
- There is no quick solutions to the problem of systematic faults in software.
- Things that could help in tolerating the same includes:
	- carefully thinking about the assumptions and interactions in the system
	- thorough testing
	- process isolation
		- run different parts of your software in separate, walled-off processes so that if one crashes or misbehaves, it doesn't bring down the others.
	- allowing process to crash and restart, since its always not possible to handle all the errors.
	- measuring, monitoring & analysing system behaviour in production

## Human Errors:
- Humans operating the system make mistakes — through typos, misunderstandings, or faulty assumptions about what an action will do — and those mistakes can cause outages or data loss. These are the errors during operations phase unlike development phase.
- The approaches that can be used to avoid the same includes:
	- Designing system in a way that minimises opportunities for errors. e.g. well-designed abstractions, APIs etc.
	- Decouple places where people make mistakes. *Sandbox* environments which are fully functional are best solutions in these cases.
	- Thorough testing: unit, integration and manual testing
	- Allow quick & easy recovery to minimise the impact e.g. fast rollback, gradual rollout of features, feature for re-computation
	- Setting up good telemetry
	- Implementing good management practices

# Scalability:
- Its the system's ability to cope with increased load. 
	- It is not a one-dimensional label. It is meaningless to say "X is scalable" or "Y doesn't scale." This means part Y of the system either needs more work/optimisation, or part Y needs to be moved entirely to a separate system to handle the specific load.
- ***Load*** can be described with few numbers called **Load Parameters** which are contextual wrt system architecture. e.g. Requests per second to a web server, ratio of reads to writes in a DB, active users in a chat room, or cache hit rate.
- Once load is defined, you investigate ***performance*** under increased load in two ways:
	- Increase a load parameter while keeping system resources (CPU, memory, bandwidth) unchanged — *How is performance affected?*
	- Increase a load parameter — *How much do you need to increase resources to keep performance unchanged?*

> [!NOTE]
> 
> - **Response Time:** What the client actually sees. It includes processing time (service time) + network delays + queueing delays.
> - **Latency:** The duration a request is waiting to be handled (latent/awaiting service).

## Percentiles & Outliers (Tail Latencies):
- Average response times are not useful for measuring outliers. You must look at higher percentiles: **p95, p99, and p999**.
	- *Example:* If the 95th percentile (p95) response time is 1.5 seconds, it means 95 out of 100 requests take less than 1.5 seconds, and 5 take 1.5 seconds or more.
- **Tail Latencies:** High percentiles are known as tail latencies. They directly affect user experience, often impacting the most valuable customers (e.g., users with the most data or purchases).
- **Head-of-Line Blocking:** A slow request in a queue blocks all subsequent requests behind it, even if those requests would individually be fast.
- **Tail Latency Amplification:** When one request triggers many parallel backend calls, a single slow backend call drags down the entire end-user response time.
## Monitoring Percentiles:
- **Naive Approach:** Keep a rolling window of response times (e.g., last 10 mins), sort the list every minute, and calculate the median/percentiles.
- **Efficient Approach:** Use approximation algorithms that maintain a **compact summary** of the distribution instead of storing every value:
	- **Forward decay** — assigns shrinking weights to older measurements so recent data dominates and stale data can be dropped.
	- **t-digest** — summarizes the distribution as a small set of cluster centroids with high accuracy at the tails (where p99 lives).
	- **HdrHistogram** — pre-allocated buckets with finer granularity at lower values; constant memory, constant-time insertion.
- *Mathematical Warning:* Averaging percentiles (to reduce time resolution or combine data from multiple machines) is mathematically meaningless. You must add the histograms together, not the averages.

## Approaches for Coping with Load:
- **Shared-Nothing Architecture:** Distributing load across multiple machines (horizontal scaling/scaling out).
- **Elastic vs. Manual Scaling:**
	- *Elastic:* Systems automatically add computing resources when load increases are detected.
	- *Manual:* Systems require human analysis to add machines (often simpler with fewer operational surprises).
- **Stateless vs. Stateful:**
	- Distributing stateless services across multiple machines is straightforward.
	- Taking stateful data systems from a single node to a distributed setup introduces massive complexity. 
- **The "Magic Scaling Sauce" Myth:** There is no generic, one-size-fits-all scalable architecture. An architecture that scales well is built entirely around assumptions of which operations will be common and which will be rare (your load parameters).
# # Maintainability:
- To minimise pain during maintenance(the costly phase of development) and avoid creating legacy software, we pay attention to three design principles:
	- **Operability:** Make it easy for operations teams to keep the system running smoothly.
	- **Simplicity:** Make it easy for new engineers to understand the system by removing as much complexity as possible. *(Note: this is about the system's internal complexity, not the user interface).*
	- **Evolvability:** Make it easy for engineers to make changes to the system in the future, adapting it for unanticipated use cases. Also known as *extensibility, modifiability, or plasticity*.

## Operability: Making Life Easy for Operations
- *Note:* This heavily translates to DevOps culture.
- Good operations can often work around the limitations of bad (or incomplete) software, but good software cannot run reliably with bad operations.
- Data systems can make routine tasks easy by:
	- Providing visibility into runtime behaviour and internals via good monitoring.
	- Providing good support for automation and integration with standard tools.
	- Avoiding dependency on individual machines (allowing a machine to be taken down for maintenance without interrupting the whole system).
	- Providing good documentation and an easy-to-understand operational model ("If I do X, Y will happen").
	- Providing good default behaviour, but allowing administrators to override them.
	- Self-healing where appropriate, but giving manual control when needed.
	- Exhibiting predictable behavior to minimize operational surprises.

## Simplicity: Managing Complexity
- As projects get larger, they often become complex and difficult to understand. A software project mired in complexity is sometimes described as a **"big ball of mud."**
- Complexity slows down developers and increases the risk of introducing bugs when making changes.
- **Abstraction:** One of the best tools for removing accidental complexity. e.g. SQL

## Evolvability: Making Change Easy
- System requirements will never remain unchanged. Unanticipated use cases, business priorities, and regulatory requirements will always emerge.
- Agile techniques (like TDD and refactoring) provide a framework for this on a local code scale, but *evolvability* applies this agility to the level of a larger data system.

# Summary:
- An application has to meet various requirements to be useful:
	- **Functional Requirements:** What it should do (e.g., storing, retrieving, searching, and processing data).
	- **Nonfunctional Requirements:** General system properties (e.g., security, reliability, compliance, scalability, compatibility, and maintainability).
